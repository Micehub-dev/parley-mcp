# Sprint 11 Brief

## Dates

- Planned window: 2026-05-08 to 2026-05-21
- Planning note: Sprint 10 closed the immediate evidence and release-operationalization gap, but the next production-readiness step is to reduce Windows-specific regression risk and further reduce manual release-review work before packaging resumes.

## Goal

- Raise Parley from a Windows-first release candidate posture into a more defensible production-readiness baseline by adding Windows automation parity, making release evidence easier to generate and review, and tightening the practical usefulness bar for Gemini in real smoke.

## Status

- In progress as of 2026-03-13

## Why This Sprint Now

- The current support boundary is still explicitly Windows-first for real operator use, but automated CI runs only on `ubuntu-latest`, which leaves the most important real-environment path underprotected against regressions.
- `npm run smoke:real` now emits useful release metadata, but maintainers still need to manually copy results into review notes and cross-check multiple docs during release review.
- Gemini now clears the shared contract and the current usefulness threshold in Windows smoke, but recent real output still falls back to the default next-step string and thin supporting detail often enough that operator usefulness remains a live production concern.
- The next bottleneck is operational confidence and repeatability, not new MCP surface area.

## Committed Scope

- [x] Add Windows CI parity for the current release-validation bar
- [x] Reduce manual release-review work by generating a reusable release-evidence artifact from the smoke path
- [x] Tighten Gemini operator usefulness without widening the shared `ParticipantResponse` contract
- [x] Refresh README, sprint, matrix, and release docs to reflect the new production-readiness baseline and residual support boundary

## Stretch Scope

- [ ] Add one lightweight release-gating check that fails clearly when expected smoke evidence fields drift
- [ ] Exercise one additional real Linux CLI verification path only if an actual Linux participant environment becomes available during the sprint
- [ ] Capture one packaging-readiness decision note only if the Sprint 11 production-readiness bar exits cleanly

## Explicit Non-Goals

- [x] no new end-user MCP tools or topic-memory features
- [x] no UI, extension, plugin, or packaging implementation
- [x] no session-state or MCP contract expansion beyond additive production-readiness hardening
- [x] no database, vector-store, or hosted control-plane work
- [x] no macOS support claim without an actual macOS runtime exercise

## Exit Criteria

- [ ] GitHub Actions exercises the core validation bar on both `ubuntu-latest` and `windows-latest`, or the repo exits the sprint with an explicit documented reason that Windows CI could not yet be added
- [x] Release review can rely on one generated smoke artifact or note payload instead of manually reconstructing launcher, environment, and usefulness facts from terminal output
- [x] Gemini remains contract-valid and more often avoids the default next-step fallback or empty supporting detail in real smoke and targeted regression coverage
- [x] Windows real-environment support wording remains honest and narrow unless an additional exercised environment is actually added
- [x] `npm test`, `npm run lint`, `npm run typecheck`, `npm run build`, and `npm run smoke:real` are green at sprint close

## Dependencies

- Preserve the current stdio-first support posture and Sprint 7 through Sprint 10 hardening behavior
- Keep diagnostics redaction, subprocess guardrails, and replay-boundary semantics unchanged unless a production-readiness gap clearly requires additive refinement
- Keep Gemini quality work confined to prompting, normalization, and release-review evidence rather than participant-specific session-state semantics

## Outcome Notes

- Sprint 11 should remain a production-readiness sprint, not a feature sprint.
- Windows automation parity is the highest-leverage next step because the current real-operator support boundary is still Windows-first.
- Release-evidence automation should reduce operator overhead without changing the MCP contract or release support claims by itself.
- Packaging direction should remain downstream of stable Windows automation, repeatable release evidence, and a stronger practical Gemini usefulness bar.
- Implementation work for the committed scope landed on 2026-03-13, but remote GitHub Actions confirmation for the new Windows lane is still pending.
- Local validation on 2026-03-13 now includes green `npm test`, `npm run lint`, `npm run typecheck`, `npm run build`, and a clean Windows `npm run smoke:real` pass after tightening Gemini fallback prompting and markdown-style normalization.
- During the same validation loop, one intermediate Windows smoke rerun still fell back to a generic Gemini identity/help response before the final two reruns passed cleanly, so the remaining production risk is response variability rather than a reproducible hard failure.
