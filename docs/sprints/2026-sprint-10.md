# Sprint 10 Brief

## Dates

- Planned window: 2026-04-24 to 2026-05-07
- Planning note: Sprint 9 closed the immediate verification, acceptance, and Gemini normalization gaps, but the product still needs stronger production-use evidence and a more repeatable release-review discipline before packaging direction resumes.

## Goal

- Raise Parley from a Windows-first production-readiness baseline into a more defensible release candidate posture by tightening real-environment evidence, improving Gemini operator usefulness, and operationalizing release evidence collection.

## Status

- Functionally complete as of 2026-03-13

## Why This Sprint Now

- The core MCP contract and runtime hardening bar are already in place, so the next meaningful risk is operational confidence rather than missing end-user features.
- The current support statement is honest but still narrow: Windows has real smoke and Codex Desktop evidence, Linux is backed by exercised CI, and macOS remains unverified.
- `npm run smoke:real` can now pass on the current Windows workstation, but recent real output still shows that Gemini may remain contract-valid while contributing low-value generic responses that reduce production usefulness.
- Release readiness is still too dependent on maintainers manually stitching together smoke, matrix, and support-boundary evidence across multiple docs.

## Committed Scope

- [x] Strengthen real-environment verification and support-boundary evidence without overstating platform readiness
- [x] Improve Gemini participant response usefulness while preserving the shared `ParticipantResponse` contract
- [x] Establish a repeatable release-evidence workflow that keeps smoke, acceptance, matrix, and support docs aligned
- [x] Refresh sprint, risk, matrix, and release docs to reflect the stronger production-use bar

## Stretch Scope

- [x] Add one lightweight release-evidence artifact template or status note format if it reduces manual release-review work without adding product surface area
- [ ] Add one additional orchestrator verification scenario only if it stays contract-neutral and does not expand the supported surface prematurely
- [ ] Capture one packaging-readiness decision note only if Sprint 10 exits cleanly against its production-use evidence bar

## Explicit Non-Goals

- [x] no new end-user MCP tools or workspace-memory features
- [x] no UI, extension, plugin, or packaging implementation
- [x] no broad contract reshaping or session-state expansion beyond additive hardening details
- [x] no database, vector-store, or hosted control-plane work
- [x] no macOS support claim without an actual macOS runtime exercise

## Exit Criteria

- [x] A real Linux CLI verification path is either exercised and documented, or the product exits the sprint with the current Windows-first support statement still explicitly intact
- [x] Linux and macOS evidence remain separated in docs and release notes rather than collapsed into a generic non-Windows claim
- [x] Gemini participant behavior remains contract-valid and shows a lower rate of low-value generic fallback output in real smoke and targeted regression coverage
- [x] The release process has one repeatable evidence path for preflight, smoke, Codex Desktop acceptance, support-boundary wording, and post-run documentation
- [x] `npm test`, `npm run lint`, `npm run typecheck`, `npm run build`, and `npm run smoke:real` are green at sprint close

## Dependencies

- Preserve the current stdio-first support posture and the Sprint 7 through Sprint 9 hardening bar
- Keep subprocess guardrails, diagnostics redaction, and replay-boundary semantics unchanged unless a production-readiness gap clearly requires additive refinement
- Prefer evidence-backed support statements over optimistic portability claims
- Keep Gemini quality work additive to the adapter and prompting layers without widening the shared participant contract
